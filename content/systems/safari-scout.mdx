## Problem Context

Planning a Kenya safari requires navigating hundreds of options across national parks, conservancies, experiences, and seasons. Most travelers spend 10+ hours researching, reading generic blog posts, and piecing together recommendations.

**The core challenge:** How do you enable intelligent, context-aware search across unstructured travel content without forcing users to read everything?

**Key constraints:**
- Data is unstructured (blog posts, reviews, guides)
- Users need personalized recommendations based on preferences
- Must handle complex queries like "3-day safari under $1000 near Nairobi with wildlife photography focus"
- Needs to work in real-time (sub-2 second responses)

## System Architecture

```
User Query → LangChain Router → Weaviate Vector DB → LLM Synthesis → Structured Response
                  ↓
           Query Enhancement
           (Budget extraction, 
            location parsing,
            preference mapping)
```

**Core Components:**
1. **Ingestion Pipeline:** Scraped + curated 2,000+ travel experiences
2. **Embedding Layer:** OpenAI embeddings stored in Weaviate
3. **Retrieval Engine:** Hybrid search (semantic + keyword)
4. **LLM Synthesis:** LangChain orchestration with GPT-4
5. **API Layer:** FastAPI with response caching
6. **Frontend:** Next.js with streaming responses

## Pipeline Breakdown

### 1. Ingestion & Embedding

**Data Sources:**

**Processing:**
```python
# Chunking Strategy
chunk_size = 500  # tokens
chunk_overlap = 50  # to preserve context at boundaries
```

**Metadata Enrichment:**

### 2. Vector Storage (Weaviate)

**Schema Design:**
```json

  "class": "TravelExperience",
  "properties": [

    {"name": "priceRange", "dataType": ["string"]},
    {"name": "activities", "dataType": ["string[]"]},
    {"name": "season", "dataType": ["string"]},

```

**Why Weaviate over alternatives?**
- **vs Pinecone:** Better hybrid search (semantic + filters)
- **vs Chroma:** Production-ready, battle-tested
- **vs pgvector:** Native vector ops, no SQL overhead

### 3. Retrieval Strategy

**Hybrid Approach:**
```python
# Semantic search for conceptual matches
semantic_results = weaviate.query.get(
    "TravelExperience"
).with_near_text({
    "concepts": [enhanced_query],
    "certainty": 0.7
}).with_limit(20).do()

# Keyword filters for hard constraints
filtered = [r for r in semantic_results 
            if matches_budget(r) and matches_location(r)]
```

**Query Enhancement:**
- Extracted entities: "under $1000" → `maxPrice: 1000`
- Location normalization: "near Nairobi" → `region: Central Kenya`
- Activity mapping: "photography" → `tags: [wildlife, scenery, golden_hour]`

### 4. LLM Synthesis (LangChain)

**Chain Design:**
```python
retrieval_chain = (
    {"context": retriever, "question": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)
```

**Prompt Engineering:**
- System: "You are a Kenya safari expert..."
- Context injection: Top 5 retrieved chunks
- Constraints: "Use only provided context. Cite sources."
- Format: Structured output (itinerary, pricing, logistics)

## Tech Decisions & Tradeoffs

### Decision 1: Why RAG over Fine-tuning?

**RAG Advantages:**
- Dynamic content updates (no retraining)
- Explainable (can trace sources)
- Lower compute cost

**Fine-tuning Disadvantages:**
- Static knowledge (outdated fast)
- Expensive retraining
- Hallucination risk without grounding

### Decision 2: FastAPI vs Flask

**FastAPI chosen for:**
- Native async support (critical for LLM calls)
- Automatic API docs (OpenAPI)
- Type safety (Pydantic models)

### Decision 3: Streaming vs Batch Responses

**Implemented streaming:**
```python
async def stream_response():
    async for chunk in llm.astream(query):
        yield chunk
```

**Impact:** Perceived latency reduced by 60% (user sees output immediately)

## Challenges & Solutions

### Challenge 1: Retrieval Accuracy

**Problem:** Generic queries returned irrelevant results.

**Solution:** 
- Added query classification (budget vs luxury, family vs solo)
- Implemented reranking with cross-encoder
- Result: 94% retrieval accuracy (measured against golden dataset)

### Challenge 2: Response Consistency

**Problem:** LLM outputs varied wildly in format.

**Solution:**
- Structured output with Pydantic schemas
- Few-shot examples in prompt
- Response validation layer

### Challenge 3: Cost Management

**Problem:** OpenAI API costs escalating ($200/month in testing).

**Solution:**
- Implemented Redis caching (70% cache hit rate)
- Switched to GPT-3.5-turbo for retrieval, GPT-4 for synthesis
- Result: $45/month in production

## What I Learned

1. **RAG is about retrieval quality, not just LLM size.** A well-tuned retriever with GPT-3.5 outperforms a naive retriever with GPT-4.

2. **Chunk size matters more than expected.** Spent 2 weeks optimizing from 200 → 500 tokens. Doubled accuracy.

3. **User queries are messy.** Built query enhancement layer to handle typos, ambiguity, implicit constraints.

4. **Streaming UX is critical.** Users tolerate 3s latency if they see progress. Same 3s feels endless with no feedback.

5. **Hybrid search wins.** Pure semantic search missed exact matches. Pure keyword missed conceptual queries. Hybrid caught both.
